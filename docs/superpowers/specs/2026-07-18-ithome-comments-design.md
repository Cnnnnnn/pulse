# IT 新闻热门评论展示设计

> 日期: 2026-07-18
> 状态: 待用户审阅
> 范围: 在现有 IT 新闻卡片正文/摘要下按需展示 IT之家热门主评论

## 1. 背景与目标

现有 IT 新闻功能支持拉取新闻列表、打开原文和按需抓取文章正文，也支持 AI 摘要，但新闻详情页的评论区尚未接入。IT之家文章页只提供评论容器，评论内容由动态脚本从评论服务加载，不包含在正文 HTML 中。

本轮目标：

- 在现有新闻卡片内提供“查看评论”入口。
- 点击后按需拉取评论，避免为每条新闻自动增加网络请求。
- 展示前 10–20 条热门主评论，字段包括昵称、内容、时间和点赞数。
- 评论加载失败时不影响新闻列表、正文、AI 摘要和原文跳转。
- 对同一篇文章缓存已成功拉取的评论，重复展开不重复请求。

## 2. 已确认的产品决策

| 决策项 | 选择 |
|---|---|
| 评论范围 | 前 10–20 条热门评论 |
| 展示位置 | 现有新闻正文/AI 摘要下方 |
| 加载时机 | 用户点击“查看评论”后按需加载 |
| 评论层级 | 仅主评论，不展示楼中楼回复 |
| 交互 | 只读，不发表评论、不点赞 |
| 原文入口 | 保留现有“阅读原文”行为 |

“前 10–20 条”实现为固定上限，建议默认取 20 条；如果接口实际只返回 10 条或更少，则展示接口返回的全部有效评论。

## 3. 架构

### 3.1 主进程评论服务

新增 IT之家评论抓取与解析模块，职责与现有 `article-page-fetcher.js` 分离：

- 根据文章链接或文章详情页信息确定评论请求所需的文章标识。
- 请求 IT之家评论服务。
- 解析评论响应为稳定的应用内结构。
- 清洗评论 HTML，保留安全的纯文本内容。
- 截取最多 20 条有效主评论。
- 返回结构化结果，并将成功结果写入对应文章缓存。

建议结构：

```text
src/main/ithome/comment-fetcher.js
src/main/ithome/comment-parser.js
```

如果实际接口验证后不需要独立解析层，则保留一个文件即可，以最少代码为准。

### 3.2 评论数据结构

单条评论统一为：

```js
{
  id: "评论唯一标识或稳定组合键",
  author: "昵称",
  content: "纯文本评论内容",
  createdAt: "原始时间字符串或标准化时间",
  likes: 0
}
```

文章缓存字段建议为：

```js
{
  comments: [...],
  commentsFetchedAt: 1710000000000
}
```

缓存只保存成功解析的评论结果。失败请求不写入空评论，避免把临时网络失败误判为“该文章没有评论”。

### 3.3 IPC 与渲染层

沿用现有 IT 新闻的 Electron 分层：

```text
NewsArticleRow
  → renderer store 的 fetchComments(id)
  → preload ithomeFetchComments(payload)
  → main IPC ithome:fetch-comments
  → comment-fetcher
  → news-store 写回缓存
```

渲染层在 `NewsArticleRow` 内维护当前卡片的评论展开、加载和错误状态；评论数据成功后通过 store signal 更新，避免把网络请求放入列表初始化流程。

## 4. 数据流与交互

### 4.1 首次点击

1. 新闻卡片显示“查看评论”，可同时显示已有评论数（如果现有新闻数据已经提供该字段）。
2. 用户点击后展开评论区域，并显示加载状态。
3. 如果文章缓存已有成功评论结果，直接渲染，不发请求。
4. 如果没有缓存，渲染层调用 IPC。
5. 主进程获取文章并请求评论服务，解析并限制最多 20 条。
6. 成功后写入缓存并返回评论，渲染评论列表。

### 4.2 再次点击

- 已加载且当前展开：收起评论区域，不重新请求。
- 已加载但当前收起：直接展开。
- 失败后点击“重试”：重新发起一次请求。

### 4.3 评论列表展示

每条评论展示：

- 作者昵称；
- 评论纯文本；
- 发布时间；
- 点赞数（有值时展示，无值时隐藏或显示 0，按接口实际字段选择一致行为）。

不渲染接口返回的原始 HTML，不执行评论中的脚本，不允许评论内容影响应用布局或插入交互元素。

## 5. 接口适配与错误处理

IT之家当前文章页的评论区通过动态脚本加载，评论接口不是稳定公开 API，且可能依赖文章级 `sn` 或动态生成参数。因此实现必须把接口失败视为可预期状态，而不是让异常冒泡影响 IT 新闻主流程。

错误分类：

- 文章不存在：`article_not_found`；
- 评论参数无法提取：`comment_params_missing`；
- 评论接口非 200、超时或网络异常：`fetch_failed`；
- 响应格式变化或没有可识别评论：`parse_failed`；
- IPC 不可用：`ipc_unavailable`。

渲染层统一显示友好文案，例如“评论暂时无法加载”，并提供“重试”。正文、摘要、收藏、已读和原文按钮必须继续可用。

接口适配约束：

- 使用现有 `HttpClient` 和超时/响应大小限制，不新增依赖。
- 仅对 `http`/`https` 的 IT之家文章和评论域名发请求。
- 不伪造登录、发表评论或点赞请求。
- 不因评论接口不可用而阻塞新闻列表刷新。

## 6. 存储与生命周期

评论跟随文章存入 `state.json.ithome_news.articles[id]`；收藏文章如果脱离当月文章缓存，沿用现有收藏快照语义，确保收藏文章的已加载评论不会因列表裁剪立即丢失。

评论缓存不参与新闻日期统计，不影响文章排序、未读角标和 AI 摘要统计。接口返回空评论时，只有在响应被成功识别且明确表示“无评论”时才允许缓存空数组；解析失败不得覆盖旧的成功缓存。

## 7. 测试策略

新增或扩展聚焦测试：

- `comment-parser`：解析真实结构化 fixture，覆盖昵称、内容、时间、点赞数、主评论筛选和 20 条上限。
- HTML 清洗：脚本、标签和危险属性不会进入最终纯文本。
- `comment-fetcher`：成功请求写入文章缓存；缓存命中不重复请求；文章不存在、非 200、超时和解析失败返回稳定 reason。
- renderer store：IPC 成功后更新评论 signal；失败保留原有新闻状态并返回错误。
- `NewsArticleRow`：未加载、加载中、成功、空评论、失败重试和收起/展开状态可渲染。

验证命令以项目现有 Vitest 脚本为准，至少运行新增评论相关测试和受影响的 IT 新闻测试；实测接口若因服务变化不可用，则使用冻结 fixture 验证解析和状态流转，并在结果中记录 live 验证限制。

## 8. 不做的事

- 不做楼中楼回复。
- 不做评论分页和全部评论加载。
- 不做发表评论、点赞、回复或登录。
- 不做隐藏 BrowserWindow 或完整浏览器脚本执行，除非直接接口方案经验证不可行且用户另行确认。
- 不新增评论搜索、筛选、排序设置。
- 不重构现有 IT 新闻列表、AI 摘要或文章正文抓取模块。
