## 1. 背景与目的

### 1.1 v1 已闭合

v1 路线图([`2026-06-19-product-roadmap-design.md`](./2026-06-19-product-roadmap-design.md))的四个 Pillar(核心升级 / 信息聚合 / 平台质量 / AI 驱动)共 31 项候选,经 v2.25 → v2.46 二十余个版本,**已 100% 落地**(见 v1 §14.3 总览刷新)。v1 §14.8 第 3 条明确建议:

> 既然主体 100% 落地,应该写一份 **v2 路线图** 替代本 v1(4 Pillar 已闭合),新方向:多端 / 团队协作 / 商业化路径。

**本 v2 即应此建议而生。v1 文档自本日起冻结**——评分表与对账附录不再维护,作为历史事实保留。

### 1.2 v2 的新命题

v2 面对的不是"接着做什么功能",而是**产品定位的边界试探**:

- v1 §8「范围之外」当时排除了 Linux / 移动端 / 云同步 / 用户账号 / 付费会员 / i18n / 自更新
- 这些边界现在是否还成立?哪些该重新打开?
- Pulse 作为**纯本地 menu bar 工具**,哪些"扩张"会削弱它的核心价值?

本文档是面向作者本人的决策辅助,**不是承诺**。核心目标:

- 把"该不该往 X 方向走"的判断**显式化**,避免凭感觉
- 每个方向标清**现实可行性**,让被排除的项有据可查(而非"忘了为什么没做")
- 给出可直接进入 `writing-plans` 的 Next 项

### 1.3 立项结论速览(决策摘要)

经 brainstorming 对齐,本 v2 立项结论:

1. **多端新平台方向整体砍掉** —— 原"多端"设想(Web/iOS/Android)与"纯本地 menu bar"定位冲突,可行性低,§8 显式收录。
2. **改为「平台纵深」Pillar** —— 把砍多端省下的精力用于打磨现有 mac/win 的获取与更新体验。
3. **团队协作方向以「防波堤」为主** —— 只留不碰账号体系的 P61(配置导入导出),其余撞 §8 红线的显式否决。
4. **商业化方向本质是「成本治理」** —— 不做付费会员,只做 BYOK 预算硬限(防 LLM 漏钱)。
5. **A5 批量编排并入 Pillar 4 收口** —— 不单立新 Pillar。
6. **A1/A2 prompt 调优不单列** —— 先建 A8 反馈闭环采数据,数据到位后调 prompt 是持续动作,非立项项。

## 2. 评分口径

### 2.1 四维评分(沿用 v1 三维 + 新增可行性)

每条候选打 4 个 0–3 的分(均为整数):

- **价值 (Value)**:对用户的真实收益。0 = 无感,3 = 解决每天都会遇到的痛点
- **成本 (Cost,反向)**:实现 + 测试 + 文档的人天。**采用 v1 §14.5 重校准刻度**:
  | 分 | 成本定义 |
  | --- | --- |
  | 0 | 跨多 Pillar + 新依赖面 + 破坏性 schema(>1000 行,需数周) |
  | 1 | 单 Pillar 内,多文件,主功能 + 测试数天可合入 |
  | 2 | 单文件或浅改(增 1 个 hook / 1 个 IPC) |
  | 3 | 1–2 行(i18n key / 文案) |
- **风险 (Risk,反向)**:回归 / 兼容性 / 用户破坏风险。0 = 高,3 = 几乎无
- **可行性 (Feasibility)** ★v2 新增:针对新方向的现实约束
  | 分 | 可行性定义 |
  | --- | --- |
  | 0 | 与产品定位直接冲突(v1 §8 明确排除)/ 无任何用户驱动 |
  | 1 | 有理论可能但需重大架构变更或新基础设施 |
  | 2 | 需新建一个子系统,但能复用现有沉淀 |
  | 3 | 基建已就位,几乎纯增量 |

### 2.2 总分公式

> **总分 v2 = 价值 + (3 − 成本) + (3 − 风险) + 可行性**,范围 0–12。

正向维(价值 / 可行性)直接加;反向维(成本 / 风险)取 `3 − x`。

> **为何新增可行性维**:v1 三个新方向(多端 / 团队 / 商业化)全部触到 v1 §8 红线。不加此维,光看价值/成本/风险会得出"商业化高价值低成本该做"的错误结论——可行性维把"与定位冲突"这层现实约束显式化。**实测效果:本维在 brainstorming 阶段直接筛掉了整个多端新平台方向**(见 §3)。

### 2.3 状态机(沿用 v1)

- 🟢 **Next**:接下来 1–2 个版本内做
- 🟡 **Maybe**:候选,看精力;评分持平或更高时可升 Next
- ⚪ **Later**:有意向但近期不做,留作远期;或已显式否决但保留评估痕迹

动工状态 5 档沿用 v1 §2.4(⚫ 未立项 / 🟤 立项中 / 🔵 实施中 / 🟢 已合入 / ⚪ 已弃用),本 v2 立项时所有 Next 项停在 ⚫ 未立项。

### 3.2 重点展开 — P52 自动更新

**动机**

现在每次发版,用户要么不知道有新版(无自检测),要么得手动去 GitHub Releases 下 dmg/exe 重装。这是 menu bar 工具最大的体验断点——它本该"常驻、即时、无感",但更新环节全是手动。

**范围(分两档落地)**

- **档 1 半自动(Next,不依赖签名)**:
  - 集成 `electron-updater`,provider 接 GitHub Releases(electron-builder 已能生成 `latest.yml`)
  - 检测策略复用现有 scheduler(`src/main/bootstrap/schedulers.js`),新增一个独立的 Pulse-self 检测通道,不与 app 检测混淆
  - UI:复用 tray 菜单 + 诊断面板(`DiagnosticsDrawer.jsx`)显示"Pulse 有新版 vX.Y.Z"
  - 下载完成后**弹窗提示手动确认安装**,不做静默替换
- **档 2 全自动(待 P51 签名落地后升级)**:
  - 后台替换 + 自动重启
  - 依赖 mac 签名(否则 Gatekeeper 拦截)

**可行性 2**:electron-updater 成熟、builder 现成;唯一外部依赖是 mac 完整无缝需签名(故分档)。

**风险 2**:更新失败 / partial download / 重启时机。缓解:复用 v1 C3 的 version-history 思路做回滚兜底;下载校验 `latest.yml` 的 hash。

### 3.3 P51 签名 + 公证(Maybe)

- **可行性 1(拖后腿项)**:需 Apple Developer 账号($99/年),是付费外部依赖,非纯工程问题。也是它只给 Maybe 的原因。
- 独立价值:消除 macOS Gatekeeper "无法验证开发者" 警告;解锁 P52 档 2。

### 3.4 P53 / P54 分发渠道

- 让用户 `brew install --cask pulse` / `winget install Pulse` 一行装上。
- `brew cask` 配 `auto_updates true` 后还能让 brew 帮忙更新,与 P52 互补。
- 价值取决于用户基数(目前未知),先 Maybe / Later,等 P52 验证自更新闭环后再评估。

### 3.5 落地建议

**P52 档 1 半自动先行**,验证自更新闭环跑通后,再决定要不要为 P51 砸 Apple Developer 费用。

### 4.1 概览

| #  | 候选                                          | 价值 | 成本 | 风险 | 可行 | 总分 | 状态     |
| -- | --------------------------------------------- | ---- | ---- | ---- | ---- | ---- | -------- |
| P61 | 配置共享文件导入导出(watchlist/提醒/持仓打包) | 2    | 2    | 1    | 3    | **9** | 🟢 Next  |

| P64 | 多用户账号 + 云端同步                          | 1    | 0    | 3    | 0    | **3** | ⚪ Later |

### 4.2 重点展开 — P61 配置导入导出(唯一 Next)

**动机**

作者自己多设备,或"把我的 watchlist / 提醒 / 基金持仓分享给同事"——这是**唯一不碰账号体系**的"协作"。本质是文件交换,不是身份系统。

**范围**

- 新增 IPC `config:export`:把以下字段序列化成一个 `.pulse-config.json`:
  - `watchlist`(v1 I2)
  - `reminders`(v1)
  - `fundPositions`(v1 I8)
  - `sidenavPrefs`(v1 I3)
  - `aiDigest.prompts`(v1 A7)用户自定义部分
- IPC `config:import`:反向加载
- **强制 diff 预览 + 字段级覆盖确认**(导入前显示"将覆盖 N 项 / 新增 M 项",冲突字段让用户勾选),避免覆盖时丢数据

**可行性 3**:state-store 全是已有字段,纯序列化 + 反序列化,无新依赖、无网络。

**风险 1**:导入覆盖旧数据 → 强制 diff 预览 + 冲突字段用户选,缓解。

### 4.3 P63 通知群推(Maybe)

- **动机**:小团队 / 家庭场景"装了同样一批 app,谁先发现更新就在群里喊一声"。
- **可行性 2**:只接 incoming webhook(企微 / 飞书 / Slack 的 incoming webhook 都是 POST JSON,无鉴权反向),**不建账号系统**。配置侧存一个 webhook URL。
- **风险 1**:webhook URL 是敏感信息 → 复用现有 `@napi-rs/keyring` 存(已用于 LLM key),不落 state.json。
- **为什么只 Maybe**:webhook 地址泄露 = 群被刷屏,作者需自担风险;且群推场景用户基数存疑。

### 4.4 P62 / P64 否决(Later)

- 直接撞 v1 §8 红线(云同步 / 用户账号)。
- P62 团队清单同步:需共享存储 + 并发控制,可行性 1。
- P64 多用户云同步:需服务端 + 账号,可行性 0。
- 列出来是为了在 §8 显式否决,避免以后反复评估。

## 5. Pillar 7 — 商业化(Monetization)

> **本 Pillar 的真实命题是「成本治理」,不是「怎么赚钱」**:Pulse 已在烧 LLM token(A1/A2/A3/A7),作者自掏腰包。商业化唯一有现实抓手的切口是"兜住正在漏的钱"。

### 5.1 概览

| #  | 候选                                          | 价值 | 成本 | 风险 | 可行 | 总分 | 状态     |
| -- | --------------------------------------------- | ---- | ---- | ---- | ---- | ---- | -------- |
| P71 | BYOK 显式化 + 用量预算硬限                    | 3    | 1    | 1    | 3    | **10** | 🟢 Next |
| P72 | LLM 用量计费看板(本地统计,非真收费)        | 1    | 2    | 1    | 3    | **8**  | 🟡 Maybe |
| P73 | Pro 订阅(高级检测源 / 更多 AI 配额)          | 1    | 0    | 3    | 0    | **3**  | ⚪ Later |
| P74 | 一次性买断 / 赞助(去通知角标 / Sponsors 链接) | 1    | 2    | 2    | 1    | **5**  | ⚪ Later |
| P75 | 企业版(团队清单 + 本地部署)                  | 1    | 0    | 3    | 0    | **3**  | ⚪ Later |

### 5.2 重点展开 — P71 BYOK 预算硬限(全场最高分)

**动机**

现在 LLM key 走 `@napi-rs/keyring` 存,但**没有用量上限**——一旦某个 app 检测触发 A1/A2 全量重算,token 消耗不可控。这是作者自用的真实痛点(钱在漏)。

**范围**

- Settings 加"每日 token 预算"输入框(默认值由作者填,如 50000/天)
- `src/ai/shared-llm.js`(LLM 调用唯一入口)在调用前查当日累计 token:
  - 超预算 → 拒绝调用 + 返回友好错误(复用 `humanizeAiError`,新增一类"配额超限"错误码)
  - 默认行为:**超限只警告不硬拒**(可配开关,避免误杀正常调用)
- token 计数:从 LLM 响应读 `usage.total_tokens`,写 `state.json.aiUsage`(每日 LRU,复用 v1 A4 的 AI 用量快照基建)
- 不碰计费、不碰支付,纯本地"防漏钱"

**可行性 3**:keyring 已有、AI 用量快照(A4)已有、shared-llm 是唯一入口。纯增量,无新依赖。

**风险 1**:预算逻辑误杀正常调用 → 默认警告不硬拒 + 预算值可随时改,缓解。

### 5.3 P72 用量计费看板(Maybe)

- 把 A4 的 AI 用量快照升级成"按模型 × 按功能的成本估算"(单价表内置),让作者直观看到"这个月 Cursor 检测花了 ¥X"。
- 价值只有 1(只可视化,不改变行为),但成本 2、可行性 3 → Maybe。

### 5.4 P73 / P74 / P75 真商业化(否决)

- 全部撞 v1 §8 红线(付费会员)。
- P73 Pro 订阅:需支付网关 + 账号,可行性 0。
- P74 一次性赞助:唯一稍有空间(Buy Me a Coffee / GitHub Sponsors 链接,不建账号、不锁功能),但价值低(不解决漏钱)、可行性 1(需作者愿意公开收款),先 Later。
- P75 企业版:需服务端 + 部署链,可行性 0。

## 6. Pillar 4 收口 — AI 编排 + 反馈闭环

> 本节收编 v1 的 A5(批量编排)和"A1/A2 盲改风险"两个遗留命题。

### 6.1 概览

| -- | ------------------------------------------------------------- | ---- | ---- | ---- | ---- | ---- | -------- |

| A9 | A5 批量编排(daily digest 升级:聚合升级建议/基金/AI 用量)   | 2    | 1    | 1    | 2    | **8**  | 🟡 Maybe |

### 6.2 重点展开 — A8 反馈闭环(化解"盲改",与 P71 并列最高分)

**动机**

v2 brainstorming 阶段作者明确判断:"没有真用户数据支撑,prompt 调整盲改风险高"。A8 就是把"盲改"变"有据改"的**前置项**——没数据就先建数据采集,再谈调优。

**范围**

- **渲染层**:
  - `src/renderer/components/UpgradeAdvice.jsx`(A2):建议结果旁加 👍/👎 按钮
  - `src/renderer/components/ChangelogSummary.jsx`(A1):加"有用 / 没用"按钮
- **信号采集(两层)**:
  - 显式:用户点的 👍/👎,记录 `{ appName, version, rec, confidence, vote, ts }`
  - 隐式:是否点了"升级"按钮 / 是否 snooze(v1 C2)/ 是否 force 重新分析(A2 现有 force 标记)——复用现有 snooze store 和 force 标记,不新增埋点
- **存储**:`state.json.aiFeedback`(LRU cap-500,复用 v1 A4 用量快照的环形结构)
- **导出**:Settings 加"导出反馈样本为 JSON"(为后续当 few-shot 源铺路)

**可行性 3**:`UpgradeAdvice.jsx` / `ChangelogSummary.jsx` 渲染点已知、state-store 是现成 LRU、隐式信号(snooze/force)全已存在。纯增量,无新依赖。

**风险 1**:反馈率可能极低(作者自用样本少)→ 这是 A8 之后接 A9 / 调 prompt 的现实约束,但**先建管道**比"等数据来了再建"合理。

### 6.3 A9 批量编排(Maybe,原 v1 A5)

**归属确认**:按 brainstorming 决定,并入 Pillar 4 作收口项,**不单立新 Pillar**。

**动机(价值 2,较 v1 A5 下调)**:A5 原设想"升级前聚合 IT / 基金 / AI 用量生成是否升级理由"。但 P52 / P61 的价值更直接,A9 的编排价值依赖 A8 数据 + A1/A2 稳定,现阶段不占优。

**可行性 2**:daily-summary-job(I5 基建)已就位,加一个"综合建议"段落可复用。但编排逻辑(A2 + 基金 + 新闻综合判断)是新代码。

**范围(若做)**:扩展 daily digest,新增 `dailyBriefingAdvice` 段——把当天所有 app 升级建议 + 基金异动 + AI 用量预警综合成一段话。

**为什么 Maybe 不 Next**:评分 8 看似不低,但**依赖 A8 先跑出数据**。A8 未落地前 A9 是空中楼阁,标 Maybe 作为"A8 之后候选"。

### 6.4 A1 / A2 prompt 调优不单列

A8 落地后,A1/A2 的 prompt 调优不再是 v2 立项项,而是 **A8 产出数据后的持续维护动作**(根据反馈样本迭代 few-shot)。不单列候选,避免与 A8 重复计分。

## 8. 范围之外(v2 显式收录)

### 8.1 继承 v1 §8(继续不做)

- **Linux 桌面端** — v2 brainstorming 评估:Electron 原生支持,detector 无平台耦合,但升级执行(brew/winget → apt/flatpak)需新适配器,且用户基数存疑。**暂不做,归 Pillar 5 远期候选。**
- **完整 i18n** — 留作远期。
- **企业版 / 团队云同步 / 多用户账号 / Pro 订阅** — 撞"纯本地"定位红线,可行性 0,显式否决(见 P62 / P64 / P73 / P75)。

### 8.2 v2 新增排除(brainstorming 显式否决)

- **整个「多端新平台」方向(Web Dashboard / iOS / Android 伴侣 App)** — v2 brainstorming 评估后整体砍掉。理由:
  - Pulse 核心价值是"常驻、即时、本地",多端要么削弱价值(Web 只读 / 移动伴侣变成另一个产品),要么需要新基础设施(跨设备同步)反而增加复杂度。
  - 可行性维在评估阶段直接筛掉(P54 / P55 可行性 0)。
  - **此决策可逆**:若未来出现明确的"非 menu bar 形态"用户场景,可重新立项。

### 8.3 保留评估痕迹(不推荐但未彻底否决)

- **P74 一次性赞助 / GitHub Sponsors** — 不锁功能、不建账号,纯打赏通道。价值低但破坏性也低,留作"作者愿意公开收款时"的轻量选项。
