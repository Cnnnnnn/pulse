## 2. 六项功能逐项规格（A–F）

> 优先级标注：**必含** = 本期必须交付；**应选** = 强烈建议；**可选** = 可延后到 P2。

### A. 稀有度 / 收藏价值分级
- **功能目标**：允许用户为每条收藏标注一个"稀有度 / 价值档位"（默认枚举 `common / rare / epic / legendary`，且允许用户重命名档位或新增自定义价值标签），并让该档位影响**排序**与**统计分布**。
- **用户故事**
  - 作为收藏者，我希望给每款游戏标一个稀有度，这样能一眼看出我的收藏里哪些"珍贵"。
  - 作为收藏者，我希望稀有度能参与排序，这样我的 legendary 能排在最前。
  - 作为收藏者，我希望统计区展示稀有度分布，这样能理解收藏的价值构成。
- **验收标准**
  - 必含：每条 `WishlistEntry` 可持有一个稀有度档位（单选、互斥）；缺省为 `unset`（null），统计/排序时 unranked 排末尾。
  - 必含：默认枚举可由用户编辑（重命名档位 / 新增自定义档位），持久化。
  - 必含：收藏列表可按稀有度降序排序（按档位 weight）。
  - 必含：统计区（StatsOverview）展示稀有度分布（各档位计数）。
  - 应选：稀有度可从 GameCard「更多菜单」与 NoteRatingModal 内设置。
  - 必含：向后兼容——旧条目无 `rarity` 字段时经 `normalizeEntry` 补 `null`，不报错、不丢数据。
- **新增 schema**：`WishlistEntry.rarity`（string|null，默认 null）；自定义档位定义 → `pulse.games.rarity.tiers.v1`。
- **依赖的 P0 能力**：① P0-3 快捷收集（GameCard 更多菜单/收藏按钮承载入口）② P0-2 tags/folders 模型（`setEntryTags` 范式为 `setEntryRarity` 参照）③ P0-5 统计（`computeCollectionStats` 扩展分布）④ P0 排序（`sortItems`）⑤ `NoteRatingModal`。

### B. 组合徽章 / 收藏成就（展示层，区别于 C）
- **功能目标**：基于现有数据（标签 / 收藏夹 / 合并主记录）**派生**一组"收藏徽章"用于展示荣耀，例如"集齐某游戏全平台""某主题满 N 款""某发行商全收"。这是**展示型荣誉**，与 C 的通用成就引擎区分。
- **用户故事**
  - 作为收藏者，我希望达成某些收藏里程碑时获得徽章，这样收藏更有成就感。
  - 作为收藏者，我希望在侧栏/专区看到我的徽章墙，这样能"秀"出来。
- **验收标准**
  - 必含：内置一组徽章规则（代码常量），对当前收藏实时求值，命中即点亮。
  - 必含：点亮徽章渲染为"徽章墙"（图标 + 名称 + 获得日期）。
  - 必含：徽章随收藏变更**响应式重算**（订阅 `wishlist` signal）。
  - 应选：徽章规则数据驱动（规则表可配置），但 P1 不开放用户自建徽章。
  - 可选：已得徽章的获得日期缓存到 `pulse.games.badges.earned.v1`。
- **新增 schema**：徽章*定义*为代码常量（不持久化）；可选缓存 `pulse.games.badges.earned.v1`（ `{id, earnedAt}` ）。
- **依赖的 P0 能力**：① P0-2 tags/folders ② P0-6 合并（`mergedMembers` 用于"全平台齐"判定）③ P0-5 统计 ④（P1a）A 稀有度（可选，丰富规则维度）。

### C. 成就系统（本地成就引擎）
- **功能目标**：构建本地成就引擎——内置 + **用户自建**成就定义、收藏变更时**自动解锁检测**、解锁历史记录。是通用"目标→检测→历史"引擎，而非 B 的固定展示荣誉。
- **用户故事**
  - 作为收藏者，我希望自定义成就（如"集齐 50 款独立游戏"），这样应用能 gamify 我的个人目标。
  - 作为收藏者，我希望条件达成时自动解锁并记录时间，这样我无需手动勾选。
- **验收标准**
  - 必含：成就定义可本地存储（内置集为代码常量 + 用户自建持久化）。
  - 必含：每个成就含一条条件规则（按 tag/folder/platform/rarity/merged 计数 + 阈值）。
  - 必含：引擎在收藏变更时自动求值，翻转 `unlocked` 并记录 `unlockedAt`。
  - 必含：解锁历史持久化（`unlocked` / `unlockedAt` / 当前进度）。
  - 应选：用户可通过弹窗自建成就（名称 + 规则维度 + 阈值）。
  - 必含：成就视图列出已锁定/已解锁，并展示进度条（复用 P0 ProgressBar 范式）。
- **新增 schema**：`pulse.games.achievements.def.v1`（用户自建定义）、`pulse.games.achievements.progress.v1`（ `{id:{unlocked,unlockedAt,current}}` ）。
- **依赖的 P0 能力**：① P0-5 `collectionStats` ② P0-2 tags/folders ③ P0-1 ProgressBar/进度条范式 ④（P1a）A 稀有度（规则维度）⑤ `collectionFilter`。

### D. 本地限时活动（纯本地调度，无后端）
- **功能目标**：纯本地调度的限时活动（如"夏季特卖收藏挑战"），按本地时钟自动生效与到期失效，进度按当前收藏计算；不联网、不拉取活动列表。
- **用户故事**
  - 作为收藏者，我希望出现限时的季节挑战，这样有新鲜动力去收集。
  - 作为收藏者，我希望活动到点自动失效，这样无需手动清理。
- **验收标准**
  - 必含：活动配置持久化（id / 标题 / `startAt` / `endAt` / 目标规则 + 阈值）。
  - 必含：内置活动为代码常量（随版本发布），用户可自建；**绝不联网拉取**。
  - 必含：处于 `[startAt, endAt]` 的活动自动浮现；过期自动隐藏至"历史/已结束"。
  - 必含：进度按当前收藏实时计算并持久化（claimed/completed 状态）。
  - 必含：到期活动锁定不可再推进，但保留历史记录。
  - 应选：仅在应用打开 / 收藏变更时重算 active/expired；展示相对时间（"还剩 3 天"），不做秒级倒计时。
- **新增 schema**：`pulse.games.events.config.v1`（活动定义）、`pulse.games.events.progress.v1`（ `{id:{claimed,completed,progress}}` ）。
- **依赖的 P0 能力**：① P0-5 `collectionStats` ② P0-2 tags/folders ③（P1a）A 稀有度（目标维度）④ `wishlist` signal 订阅 ⑤ 本地时钟（无 IPC）。

### E. 本地使用埋点（纯本地事件计数，不上传）
- **功能目标**：纯本地的关键行为计数器（收藏次数 / 打标次数 / 合并次数等），用于个人回顾；**无上传、无后端、无账号**。
- **用户故事**
  - 作为用户，我希望看到自己如何使用收集功能（收藏了多少、建了多少标签），用于自我回顾。
  - 作为产品方，我希望有轻量计数理解参与度（纯本地，合规零风险）。
- **验收标准**
  - 必含：计数器表持久化（`eventName → count`，含 `firstSeen`/`lastSeen` 时间戳）。
  - 必含：在关键 action 处**作为副作用**自增：addToWishlist / removeFromWishlist / setEntryTags / createFolder / mergeEntries / splitEntry / setRating / setNote /（P1a）setRarity。
  - 必含：提供"使用回顾"面板展示计数；**任何路径都不触发网络**。
  - 必含：UI 明确标注"仅本地，不上传"。
  - 应选：仅计数 + 时间戳，**不记录逐条明细**（隐私最小化）。
- **新增 schema**：`pulse.games.metrics.v1`（ `{event:{count,firstSeen,lastSeen}}`，或简版 `{event:count}` ）。
- **依赖的 P0 能力**：① P0 store actions（作为计数器埋点钩子，零新引擎）② `wishlist` signal。

### F. 分享图生成（纯本地 canvas，不依赖后端）
- **功能目标**：把收藏墙 / 统计 / 徽章 / 成就渲染成图片并本地导出（canvas → PNG），纯本地、不依赖任何后端图片服务。
- **用户故事**
  - 作为收藏者，我希望导出一张好看的收藏墙图片，分享到社交平台。
  - 作为收藏者，我希望能选布局/模板。
- **验收标准**
  - 必含：「生成分享图」按钮将当前收藏（缩略色块 + 标题 + 统计 + 徽章/成就）绘入 `<canvas>`。
  - 必含：导出 PNG——优先 Electron `saveDialog` 指定路径保存；失败降级为锚点下载。
  - 必含：至少 1 个默认模板；用户可选项布局存 `pulse.games.share.templates.v1`。
  - 必含：渲染用系统字体 + `oklch()` 色（不引外部资源）；尊重 `prefers-reduced-motion`（静态渲染）。
  - 必含：纯本地 canvas，无网络、无后端图片服务。
  - 应选：缩略图安全模式——P1 先支持"色块 + 标题"安全渲染；远程缩略图增强经主进程 IPC 下载为本地 dataURL 后再绘（避开 canvas 跨域污染）。
- **新增 schema**：`pulse.games.share.templates.v1`（用户布局偏好；默认模板为代码常量）。
- **依赖的 P0 能力**：① GameCard `thumb` ② StatsOverview ③（P1b）B 徽章 /（P1c）C 成就（丰富图像内容）④（P1a）A 稀有度 ⑤ 主进程图片下载 IPC（既有通道，非新增）。

## 3. 推荐实现顺序 / 批次化（P1a / P1b / P1c）

### 3.1 分批原则
- **P1a — 数据增强 & 轻量埋点**：纯加法 schema 扩展、零新运行时引擎、回归风险最低 → 最先交付，验证"加字段 + 持久化 + 派生"范式。
- **P1b — 展示与导出层**：基于 P0 + P1a 数据做视图/导出，无调度引擎。
- **P1c — 本地引擎**：成就引擎 + 限时活动调度，复杂度最高，需前面批次的数据基础才有意义。

### 3.2 批次 — 功能 — 依赖 总览

| 批次 | 功能 | 类型 | 风险 | 依赖的 P0 能力 | 本批理由 |
|---|---|---|---|---|---|
| **P1a** | **A 稀有度分级** | 数据增强 | 低 | P0-2/3/5 + sortItems + NoteRatingModal | 加法字段 + 统计/排序扩展，零引擎 |
| **P1a** | **E 本地埋点** | 副作用计数 | 最低 | P0 store actions | 仅计数器，无新 UI 引擎，零基建 |
| **P1b** | **B 组合徽章** | 展示派生 | 低–中 | P0-2/5/6 +（A） | 基于现有数据派生荣誉墙，无调度 |
| **P1b** | **F 分享图** | 导出 | 中 | GameCard/StatsOverview +（A/B/C） | canvas 渲染 + 导出，UI 较重但有界 |
| **P1c** | **C 成就系统** | 本地引擎 | 中–高 | P0-1/2/5 +（A） | 通用"定义→检测→历史"引擎 |
| **P1c** | **D 限时活动** | 本地调度 | 中–高 | P0-2/5 +（A）+ 本地时钟 | 时间窗 + 自动失效调度 |

## 6. localStorage Schema 扩展汇总

## 7. P1 专属待确认问题（仅 P1 新增，附推荐默认决策）

| # | 待确认问题（P1 新增） | 推荐默认决策 |
|---|---|---|
| Q1 | 稀有度默认值语义：所有条目是否自动归 `common`，还是允许 `unset`？ | **允许 `unset`（null）**；统计/排序时 unranked 排末尾；提供"批量设为 common"入口。 |
| Q2 | 稀有度是否影响"估值"金额统计（如 legendary 加权）？ | **不影响金额**（保持 `currentPrice` 真实）；仅影响分布展示与排序。 |
| Q3 | 自定义价值标签与既有 `tags` 是否二义？ | **明确区分**：rarity 单选、互斥；tags 多选、自由。不合并。 |
| Q4 | B（徽章）与 C（成就引擎）在 P1 是否强制合并？ | **不强制**：B 作为轻量展示层（规则=代码常量）独立实现；C 为通用引擎。后续可把 B 规则注册进 C。 |
| Q5 | C 成就条件支持哪些维度？ | P1 支持：按 tag / folder / platform / rarity / merged(全平台齐) 计数 + 阈值；**暂不支持**价格阈值、时间窗等复杂条件。 |
| Q6 | D 内置活动由谁定义、如何更新？ | **内置活动 = 代码常量**（随版本发布）；用户可自建；**不联网拉取**（遵守无后端约束）；结束后保留历史。 |
| Q7 | D 是否需要秒级实时倒计时？ | **不需要**：仅在应用打开 / 收藏变更时重算 active/expired；展示相对时间（"还剩 3 天"）；如需倒计时再议。 |
| Q8 | E 埋点是否记录逐条明细 / 仅计数？ | **仅计数 + firstSeen/lastSeen 时间戳**，无逐条明细；纯本地，明确不上传。 |
| Q9 | F 远程缩略图绘入 canvas 会跨域污染（taint）导致 `toBlob` 失败，如何处理？ | P1 先支持**"色块 + 标题"安全模式**；远程缩略图为增强——经主进程 IPC 下载为本地 dataURL 后再绘。 |
| Q10 | F 导出方式：Electron `saveDialog` 还是直接下载？ | **优先 `saveDialog` 指定路径保存 PNG**；失败降级为锚点下载。 |

## 8. 非目标 / 明确不做（P1 范围外）

- 不引入后端 / 账号 / 云同步 / 任何网络出口。
- 不做社交平台直传（分享仅生成本地图片文件）。
- B 与 C 不在 P1 强制统一为同一引擎。
- 不做价格自动刷新（沿用 P0 `refreshWishlistPrices` 的 no-op 约定）。
- 不做跨设备迁移工具（localStorage 本地优先，P2 再议导入/导出整个收藏库）。
