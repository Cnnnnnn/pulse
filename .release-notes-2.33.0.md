## 新增
- **微博热搜未读角标**:SideNav「🔥 微博热搜」item 右上角显示红色数字胶囊,
  数字 = 本 session 新增且还没读的热搜词数 (与 v2.32.0 的 ithome 角标对称)
  - 点开一个热搜词(打开微博) → 行变灰 + badge -1
  - refresh 拉到新词 → badge 增量
  - 重启 → 行仍变灰(readIds 持久化), badge 归 0(newIds session 级)
  - 与 ithome 角标**各自独立**显示,互不影响
- **微博热搜「已读」持久化**:点击过的热搜词记录到 `state.json.wechat_hot.readIds`,
  跨重启保留 (此前 wechat-hot 完全无持久化,本次从零补齐)
- **SideNav 双面板 badge**:`navBadges` 现在同时驱动 ithome + wechat-hot 两个角标,
  SideNavItem 的通用 badge prop 扩展点兑现

## 范围说明 (v1)
- 严格限定:**只做 wechat-hot 一个面板的 read + badge**。
  - 仿 ithome 完整 read 机制:main read-store → state-store/schema → IPC →
    renderer store → WechatHotList 行级已读 → SideNav navBadges。
  - diff key = `title`(热搜词本身;rank 随热度浮动不稳定)。
  - 点行 openExternal(打开微博)顺带标记已读,零额外点击。
- 不做收藏 / 摘要 / 分享(ithome 有但 wechat-hot 不需要);
  不做 "全部已读"按钮;readIds 不设容量上限(热搜词轮回,distinct 词数有限)。

## 技术细节
- **read-store.js**:仿 ithome `news-store.js` 的 `markArticleRead`,用
  `stateStore.patchState` 做 atomic write,保留已有 readIds + 其它 state 字段。
- **diff key = title**:`rank` 随热度浮动不稳定;`url` 由 word 生成也可,但 `title`
  是热搜词的真实身份,最直接可读。
- **bootstrap 先拉 readIds 再 load**:diff 依赖 readIds 判断"已读",所以 bootstrap
  开头先 `api.wechatHotLoadRead()` 拉回持久化的已读词。
- **TDD 暴露并修复 2 处回归**:全量测试发现 `store.test.js` /
  `wechat-hot-layout.test.jsx` 两个已有测试文件的 mock 缺新增导出,已补上。

## 风险

| 风险 | 等级 | 缓解 |
| --- | --- | --- |
| title 微变(微博加 emoji/标点)导致 diff 不准 | 低 | 热搜词本身较稳定;不准也只是 badge 数字偏差,不影响主功能(行已读/打开正常) |
| readIds 无限累积撑爆 state.json | 极低 | 热搜词轮回,distinct 词数有限;必要时加 LRU cap(v3) |
| markItemRead 与 openExternal 时序 | 无 | markItemRead 乐观更新 + fire-and-forget IPC,不阻塞跳转 |
| bootstrap 多一次 IPC 往返(load readIds) | 极低 | 非关键路径,一次额外往返可接受 |
