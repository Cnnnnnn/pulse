## 新增

- **诊断面板"启动时间"区块接通** (修 P0 数据丢失 bug)
  - Q1 v2 暴露了 `markBootstrapDone` / `markRendererReady` 两个里程碑 setter,
    但全代码库 0 调用点 (grep 验证),导致抽屉"启动时间"区块永远显示 `-`
  - 本次:
    - `src/main/index.js` 顶部 require `./diagnostics` → t0 被采
    - `markBootstrapDone()` 在 `whenReady → bootstrap()` 末尾触发
      (tray / window / pool / ipc 全部就绪后)
    - `markRendererReady()` 在 renderer `did-finish-load` 触发
  - 效果: 抽屉 bootstrap / readyMs 显示真实毫秒;
    `state.json startup_samples` cap-20 环式累积,长跑可看趋势
- **冷启动 baseline profiler** — `npm run baseline:q4 -- --runs=N`
  - 测 main 进程 require 链每个模块的冷加载 ms
  - v1 实测 (5 跑取中位数):
    - 全部 25 模块 require 总耗时 **16.1ms** (range 14.9..55.3ms)
    - `src/main/ipc` 最大 **9.9ms** (受 OS 调度噪声主导,非代码瓶颈)
    - 23 个非 ipc 模块 ≤ 1.0ms
  - 给 v2 优化做锚点,回归检测用

## 优化

- **零** (经过 review)
  - `src/main/ipc/register-core.js` 顶部 require 链 12 个模块,经逐个检查

  - 但 `../log` 已被 `index.js` 顶部 require 过一次,register-core 顶部走 module cache,

## 风险

| 风险 | 等级 | 缓解 |
| --- | --- | --- |
| `markRendererReady` 在 `did-finish-load` 触发但 preload 没完全 ready | 低 | did-finish-load 是 renderer 完整加载完 (含同步 script),spa 启动后所有 dom ready 信号都行 |
| 在 vitest 环境下 require diagnostics 会采 t0,污染测试 | 低 | vitest 不 require index.js,直接跑 diagnostics.test.js 仍然 fresh;且 diagnostics 自身 _resetForTest 可清状态 |
| 真实 baseline > 800ms | 中 | 留 v2 优化池 (webpackChunks / dynamic import / preload 拆分);v1 先打通 telemetry |
